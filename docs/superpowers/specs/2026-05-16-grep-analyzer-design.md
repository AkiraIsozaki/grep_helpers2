## 1. 目的（PRDの再掲・要約）と網羅性の約束

Java / C / Pro\*C / SQL / Shell（v1）が混在する大規模レガシーシステムで、特定の文言（文字列値・コード値）を転用・削除しようとする開発者が「その値が今どこでどのように使われているか」を把握できるようにする。grepで拾えない**定数経由・getter/setter経由・変数経由の間接参照**まで追跡し、**使われ方を自動分類**し、**横にひと目でわかる要約**を付けたTSVを出力して、数万行規模の手作業目視確認を不要にする。

### 網羅性の約束（PRDからの誠実化・重要）

純Python・javalang（シンボル解決を持たない）という構成上、「**1件の見落としもなく / ゼロ**」という絶対保証は構造的に成立しない。本ツールの約束を次に置き換える:

> 追跡ポリシー（§8）の範囲で**到達可能な間接参照を不動点まで最大限追跡**し、grep単独より網羅性を大幅に向上させる。原理的に追跡不能なクラス（§8.4）は**「既知の限界」として diagnostics に明示**し、レビュアーが見落としを認識できる状態にする。

> ⚠ PRD 本文の「1件の見落としもなく」「取りこぼしをゼロに」は本 spec と不整合。**PRD 側の文言を本節に合わせて要修正**（フォローアップ）。

### 4.3 フィージビリティ・スパイク（writing-plans 着手前のブロッキング前提）

トラックB実装計画の確定前に、実機で以下を**ゲートとして**検証する。

**成功基準（すべて満たすこと）:**
1. **G1**: `cc -V` ≥ 12.4（C11 完全サポート）。
2. CPython（後述バージョン）が `cc`＋`/usr/sfw/bin/gmake` でビルドでき、起動する。
3. `zlib`/`ctypes` import 成功（`ssl` は任意・既定で要求しない）。
4. `pip install --no-index --find-links wheelhouse javalang chardet` が**推移依存 `six` 込みで**成功（`pip --version` がオフラインで動く＝ensurepip展開成功を含む）。
5. **代表 Java ファイル群を javalang に通し、パース成功率を実測**（Java 11-17 構文の失敗率を把握。§7 の精度前提の検証）。
6. **現地 SPARC で 1GB の単一ストリーミング走査スループットを実測**（§8.2 の総時間外挿のゲート）。

**フォールバック階梯（G1 不成立 = Studio < 12.4 の場合・確定的に分岐）:**

| 実機 Studio | ビルド対象 CPython | トラックB言語制約 |

| ≥ 12.4 | **3.12.x**（既定） | 3.12 構文をフル使用可 |

## 5. トラックB：アーキテクチャ

### 5.1 パイプライン

```
Ingest → Seed化 → 言語ディスパッチ → 分類器 → 不動点・ターゲットスキャン・エンジン → TSV出力(+診断)
```

- **Ingest**: `input/*.grep` を文字コード判定の上で読込・行パース（§6）。
- **Seed化**: `{keyword, source_path, lineno, raw_line, language, match_span}` のseed集合。
- **言語ディスパッチ**: 拡張子/ヒューリスティックで言語モジュール選択。
- **分類器**: §7。
- **走査エンジン**: §8。
- **TSV出力**: §9。診断は別ファイル（§10.3）。

### 5.2 言語別プラグイン・インタフェース（v2+ 追加点）

各言語モジュールは共通インタフェースを実装:

- `classify(line, file_context) -> Category`
- `extract_chase_symbols(seed_or_hit) -> set[Symbol]`（不動点拡張のため、ヒットからも再抽出可能であること）
- `tracking_policy`（§8.3）

## 6. 入力契約

- `input/<keyword>.grep`（複数可。ファイル名 stem = キーワード）。
- 1行 = `grep -rn` 相当 `path:lineno:content`。**パース規則を厳密化**:
  - 正規表現 `^(?P<path>.*?):(?P<lineno>\d+):(?P<content>.*)$` を用い、**最左の `:<数字>:` 境界**を採用（非貪欲）。
  - これにより Windows パス（`C:\...`）・content 先頭が数字+`:`・パスに `:` を含むケースの誤分割を回避。
  - 一致しない行は**黙って捨てず** diagnostics（§10.3）へ。
- 別途 CLI `--source-root <60GBツリー>` を tool が走査。
- `.grep` 自体も文字コードバラバラ前提 → 入力もエンコーディング判定対象（§10.1）。

## 7. 分類器（言語別）

`category` は「使われ方」の機械判定。`confidence` の意味を**厳密化**:

- `high` = **AST により構文位置が確定**（if条件/return/引数 等の構文要素が確実）。**意味的同一性（その`x`が本当に当該値を持つ型か）は保証しない**。
- `medium` = 正規表現分類。
- `low` = フォールバック、または型解決を要する間接（getter/setter 等）で人間確認前提。

言語別:

- **Java**: 対象コードに **Java 11〜17 構文を含む**ため、**正規表現分類を堅牢な主軸**とする（confidence=medium）。javalang AST はファイルがパースできた場合のみ適用し精度向上（confidence=high）。Java 11-17 構文（var/record/switch式/text block 等）で javalang パース失敗時は regex に縮退（medium）。判定軸: 宣言 / 比較（if・equals）/ 代入 / メソッド引数 / return / switch-case / アノテーション / 文字列連結 / ログ・出力。
- **C / Pro\*C**: 正規表現。`#define` / 変数初期化 / 比較 / 関数引数 / `switch case` / 文字列。Pro\*C は `EXEC SQL` ホスト変数を追加。
- **SQL**: 正規表現。リテラル比較 / `INSERT`・`UPDATE` 値 / `CASE WHEN` / 変数代入。
- **Shell**: 正規表現。変数代入 / 比較（`[ "$x" = "X" ]`）/ `case` / コマンド引数。

`category` は **二層構造**: 言語横断の共通上位カテゴリ（比較 / 代入 / 引数 / 宣言 / 出力 / 分岐）＋ 言語固有サブカテゴリ。これにより Excel フィルタが言語をまたいで機能する（PRDの主目的）。サブの語彙詳細は実装計画で言語別に確定。

## 8. 不動点・ターゲットスキャン・エンジン（網羅性優先）

ユーザー決定: **網羅性を上げるため追跡を強化し、性能犠牲を許容する**。

### 8.1 アルゴリズム

1. seed/既存ヒットから各言語モジュールが「追跡シンボル」を抽出（リテラルを束ねる定数識別子、連鎖代入先、getter/setter名、SQL/Shell の変数代入）。
2. シンボル集合を**単一の結合マッチャ**に変換（純Python。大規模時は純Python実装 Aho-Corasick、小規模は連結正規表現。閾値は実装計画で決定）。
3. ソースツリーを **multiprocessing でストリーミング走査**（バイナリ/巨大ファイル skip、include/exclude glob、行単位ストリーム）。
4. 新規シンボルが出れば集合に追加し**不動点（新規が出なくなる）まで反復**。多ホップ定数・連鎖代入も追う。
5. **暴走防止ガード**（性能犠牲を許容しても無限/爆発は防ぐ）: シンボル循環検出、設定可能な安全上限 `--max-depth`（既定は十分大きく、超過時は diagnostics に明示して打ち切り）、シンボル集合サイズ上限（超過時は degrade してマッチャ分割＋警告）。
6. 各間接ヒットを分類し、起点seedへの**来歴チェーン**を記録。

### 8.2 性能設計（段数 ≠ 走査回数）

**オーダー見積り（前提の明示）**: 旧 SPARC 上の純Python走査スループットを概ね **5〜20 MB/s** と想定すると、60GB の単一フルパスで **約1〜3.5時間/パス**。不動点反復で素朴に毎回フルパスすると 1キーワード半日〜1日級。ユーザーは性能犠牲を許容するが、実用維持のため**走査回数を段数に比例させない**設計とする:

- **2フェーズ＋遅延シンボル成長**: フェーズ1で「現在の全シンボル」を1パス走査して全ヒットと、ヒット行から派生する新シンボル候補を収集。新シンボルは**次フェーズへバッファ**し、フェーズ2で追加シンボルのみを対象に再走査。これを新規が枯れるまで反復するが、各反復は**シンボル集合の差分のみ**を対象にできるよう、ファイル→該当行のローカリゼーション情報を保持して全60GB再読込を避ける（メモリ上限内。超過時は degrade）。
- 走査総量の上限は実装計画で「最悪 N パス」を明示（N は max-depth と差分戦略で有界化）。
- `multiprocessing.Pool` 並列度＝検出コア数。タスク粒度＝ファイル単位、ワーカーは結果（ヒット＋来歴断片）のみ返却（pickle 量を抑制）。
- ファイルサイズ上限＋バイナリ判定 skip。結合マッチャ事前コンパイル。進捗/ETA を stderr。
- **メモリ上限**: 設定可能。超過時はマッチャ分割・中間結果のディスクスピルで degrade（クラッシュさせない）。
- 実時間の妥当性は §4.3 の現地1GB実測から外挿して**ゲート判定**する。

### 8.3 追跡ポリシー（不動点・網羅優先）

旧「言語別の小さな固定段数打ち切り」は**廃止**。代わりに:

- 全言語: 直接参照 ＋ 間接参照を**不動点まで多ホップ追跡**（定数→定数、連鎖代入、Java/Groovy は getter/setter 経由も含む）。
- 終了条件: 新規シンボルが出ない（不動点）／`--max-depth` 到達（到達時は diagnostics 明示）／安全上限到達。
- Shell/SQL の同一ファイル内変数、C/Pro\*C の `#define`・連鎖代入もホップ対象。

### 8.4 既知の取りこぼしクラス（原理的限界・diagnostics に必ず明示）

追跡を最大化しても、本構成では**原理的に追えない**クラスが残る。これらは取りこぼしたら黙らせず diagnostics に列挙し、レビュアーが認識できるようにする:

- **型解決を要する getter/setter**: javalang はシンボル/型解決を持たないため、`obj.getX()` の `obj` の型同定は不可。同名メソッドは全件 `confidence=low` で出すが意味的同一性は人間判断。
- **リフレクション / 動的呼び出し**（`Class.forName`、`Method.invoke` 等）。
- **実行時に文字列連結・整形で生成される名前**（`"PREFIX_" + x`、フォーマット生成）。
- **生成コード**（`target/`・`gen/` 等）: include/exclude のデフォルトポリシーを実装計画で定義し、除外時は「除外した旨」を diagnostics に記録。
- **javalang パース失敗（Java 11-17 構文）箇所**: regex 縮退（confidence 低下）した旨を記録。

## 9. TSV 出力スキーマ

`output/<keyword>.tsv`（キーワード1件＝1ファイル）。タブ区切り、フィールド内タブ/改行はサニタイズ。出力文字コードの既定は **UTF-8 with BOM**（Excel が文字化けせず開ける）。`--output-encoding`（例 `cp932`）で変更可。

| 列 | 内容 |
|---|---|
| `keyword` | 調査対象の文言 |
| `language` | 判定言語 |
| `file` | ファイルパス |
| `lineno` | 行番号 |
| `ref_kind` | `direct` / `indirect:constant` / `indirect:getter` / `indirect:setter` / `indirect:var`（言語別マッピングは下表） |
| `category` | 共通上位カテゴリ（§7） |
| `category_sub` | 言語固有サブカテゴリ |
| `usage_summary` | ★ひと目要約（例:「定数 STATUS_OK 経由で if 比較」）= PRDの「横にひと目でわかる単位」 |
| `via_symbol` | 経由シンボル（direct は空） |
| `chain` | 来歴（起点→経由→ヒット。多ホップ可） |
| `snippet` | 該当行（タブ/改行除去） |
| `encoding` | 判定文字コード（置換発生時「要確認」フラグ） |
| `confidence` | `high`(AST構文位置) / `medium`(regex) / `low`(型解決依存・fallback) |

**ref_kind × 言語マッピング（一意化）:**

| 言語 | constant | var | getter/setter |
|---|---|---|---|
| Java | `static final` 定数 | 非finalフィールド/ローカル | `indirect:getter`/`indirect:setter` |
| C/Pro\*C | `#define`・`const` | 非const変数 | — |
| SQL | — | 同一ファイル変数代入=`indirect:var` | — |
| Shell | readonly等 | 変数代入=`indirect:var` | — |

**決定性（golden の前提・必須）**: 出力は **(file, lineno, ref_kind, via_symbol, chain) で安定ソート**。`multiprocessing` の完了順に依存しない。

**規模対策**: 間接展開で TSV が数十万行に達し得る。1キーワードの行数が Excel 上限（約104万行）を超える見込みなら、`<keyword>.partNN.tsv` に分割し diagnostics に警告。

## 10. 横断事項

### 10.1 文字コード

- バージョン固定 `chardet`（§4.1）で判定 → 決定的フォールバック鎖: `utf-8` → 検出結果 → `cp932`/`euc-jp` 候補 → `latin-1`（最終手段・置換）。
- **decode で絶対に落とさない**。置換発生時 `encoding` 列＋diagnostics に「要確認」。
- chardet をバージョン固定する理由: 同一バイト列に対し判定が決定的でないと golden（完全一致）が壊れるため。
- 入力 `.grep` と走査対象ソースの両方に適用。
- → PRDメモ「文字コード判定はあったほうが安全」を解決。

### 10.2 性能

§8.2。

### 10.3 エラー処理・中断

- 壊れたgrep行 / 読めないファイル / decode失敗 / javalangパース失敗 / `--source-root` 不在 / 権限拒否 / max-depth到達 / 生成コード除外 → すべて **diagnostics ファイルへ集約**（非致命）。部分成功でも使えるTSVを出す。
- **キーワード単位コミット**: 1キーワード分の処理完了ごとに当該 TSV を確定保存。半日級実行（§8.2）の途中中断でも処理済みキーワード分は保全。
- 終了コードで「致命」と「警告のみ」を区別。

### 10.4 CLI

```
grep_analyzer --input ./input --output ./output --source-root <tree> \
  [--jobs N] [--include GLOB] [--exclude GLOB] [--max-depth N] \
  [--memory-limit MB] [--encoding-fallback ...] [--output-encoding cp932]
```

- `input/` に複数 `<keyword>.grep` 可。出力はキーワード毎 TSV ＋ 全体 diagnostics 1ファイル。

## 11. テスト方針（リポジトリ規約準拠）

`.claude/skills/writing-tests.md` に従う（古典学派 + TDD、日本語テスト名、層の縄張り）。

- `tests/unit/`: 分類器・追跡シンボル抽出・grep行パース（Windowsパス/数値始まりcontent/`:`含有パス）・文字コードフォールバック・不動点終了条件。
- `tests/integration/`: CLI契約・複数keyword・エラー経路・中断/再開挙動・オプション相互作用（in-process 既定）。
- `tests/golden/`: **合成された小規模代表ツリー**（各言語 × 各 ref_kind × 文字コード混在 × 多ホップ × javalang失敗ケース の最小事例を意図的に構築。git 管理可能サイズ）→ TSV 完全一致。決定性ソート（§9）が前提。
- `tests/handcrafted/`: 分類精度ダッシュボード（pass/fail ゲートではない）。
- `tests/test_smoke.py`: `import` ＋ `--help` exit 0。
- **性能は golden/CI で検証しない**。性能妥当性は §4.3 の現地スパイク実測でのみゲート判定（本物60GBは持ち出せない前提）。
- 本ツールに LLM は無いため規約のLLM項は N/A。外部I/O境界（ファイル/サブプロセス）は本物。

## 12. スコープ外・却下した代替案

- **soot**: JVM 依存。オフライン Solaris 10 SPARC で高リスク。Java AST 需要は pure-Python の `javalang` で（限定的に）充足。→ 不採用。
- **tree-sitter**: ネイティブ拡張ビルド必要。SPARC prebuilt 無し、現地 cc ビルドは CPython 同等の高リスク。→ 不採用。
- → PRDメモ「sootやtree-sitterも有効活用できないかな？」への回答。
