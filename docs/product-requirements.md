## プロダクト概要

### 名称
**grep_analyzer** - grep結果 自動分類・使用箇所洗い出し＆見やすく配置するツール（v1対応言語: Java / C / Pro\*C / SQL / Shell。v2+ で Kotlin / PL/SQL / TypeScript・JS / Python / Perl / C#・VB.NET / Groovy を順次追加）

### プロダクトコンセプト
- **網羅的な使用箇所洗い出し**: grepでヒットした文言の直接参照だけでなく、定数/変数/getter/setter経由の間接参照まで**不動点（新たな経由が見つからなくなるまで）多ホップで追跡**し、grep単独より網羅性を大幅に向上させる。型解決を要するgetter・リフレクション・実行時生成名など**原理的に追跡できないクラスは「既知の限界」として診断出力に明示**し、レビュアーが見落としを認識できる状態にする（絶対的なゼロ見落としは構造上保証しない）。
- **使われ方の自動分類**: 言語ごとの使用パターンを自動判定し、人間の目視確認コストを大幅に削減する。**構文解析の主軸は tree-sitter**（Java は 8〜17+ を正確にパース。Pro\*C は tree-sitter-c ＋ EXEC SQL 前処理で補完。SQL / Shell は正規表現）。
- さらに、その結果の横にひと目見ただけでそれがどのように使われているのかがわかる単位（`usage_summary` 列）での出力を行う。

### プロダクトビジョン
Java / C / Pro\*C / SQL / Shell（v1）が混在するプロジェクト内の特定の文言（文字列値・コード値）を転用・削除しようとする開発者が、「この値は今どこでどのように使われているか」を、追跡ポリシーの範囲で**可能な限り網羅的に**把握できるようにする。grepだけでは拾えない定数経由・getter/setter経由の間接的な使用箇所まで追跡し、TSV形式で一覧化することで、数万行の手作業確認を不要にする。

### 目的
- grep結果4万行規模の手作業目視確認を自動化する。grep対象の資源は合計で60GB以上あるため、パフォーマンス面の考慮も重要（モダンLinux＋ネイティブ手段で実用時間内に走査）。
- 言語ごとの使われ方パターンに自動分類し、影響範囲の把握を容易にする。結果の横に、ひと目でわかる粒度の要約を並べる。
- 直接参照だけでなく間接参照（定数/変数/getter/setter経由）まで不動点まで追跡し、転用・削除時の取りこぼしを最小化する。追えない原理的限界は診断に明示する。
- TSVを出力し、チームでの調査共有を容易にする。

## ターゲットユーザー

### プライマリーペルソナ: Java・C・Pro\*C混在レガシーシステムのリファクタリング担当者

- 大規模プロジェクト（数千〜数万ファイル、Java/C/Pro\*C/SQL/Shell混在）を保守・改修している
- 特定の文字列値・コード値（ビジネスコード・ステータス値など）の意味を変更・廃止したい
- その文字列が現在どこでどのように使われているかを可能な限り網羅的に調査する必要がある
- **現在の課題**: grepで4万行ヒットしても1行ずつ確認するのは現実的でない。定数経由の間接使用はgrepでは拾えず見落としリスクがある（IDEの呼び出し検索でも可能だが大変なので避けたい）
- **期待する解決策**: 自動分類したTSVを受け取り、Excelでフィルタしながら方針を決めたい。横にひと目で理解できる粒度の要約も並んでいると嬉しい
- **典型的なワークフロー**: grepを手動実行 → input/に配置 → ツール実行 → output/のTSVをExcelで確認

## 確定事項（設計書で解決済み）

旧「メモ(聞いたこと)」の各論点は以下の通り確定（詳細は設計書・最新版）:

- **稼働環境**: 稼働先は **モダンLinux・オフライン**。Solaris 10 で動かす案は取りやめ。開発＝稼働ともに **Python 3.12**（旧「3.7環境での稼働」「Solaris自前ビルド」は撤廃）。依存は prebuilt manylinux wheel を wheelhouse に同梱し `pip --no-index` で導入。
- **文字コード判定**: 入力ファイルの文字コードがバラバラな前提で、判定＋決定的フォールバック鎖を実装（決して落とさず、置換発生は診断に明示）。
- **tree-sitter / soot**: **tree-sitter を構文解析の主軸として採用**（Java/C/Pro\*C基盤）。**soot は不採用**（JVM依存、tree-sitterで代替）。
- **入力ファイル名**: `<キーワード>.grep`（input/ に配置）で確定。1行は `grep -rn` 相当 `path:lineno:content`。

詳細な制約・アーキテクチャ・TSVスキーマ・リスクは設計書 `docs/superpowers/specs/2026-05-16-grep-analyzer-design.md` を参照。
