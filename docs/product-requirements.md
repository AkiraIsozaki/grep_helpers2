# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称
**grep_analyzer** - grep結果 自動分類・使用箇所洗い出し＆見やすく配置するツール（Java / C / Pro*C / SQL / Shell / Kotlin / PL/SQL / TypeScript・JS / Python / Perl / C#・VB.NET / Groovy 対応）

### プロダクトコンセプト
- **網羅的な使用箇所洗い出し**: grepでヒットした文言の直接参照だけでなく、定数/変数/getter/setter経由の間接参照まで追跡し、1件の見落としもなく全件洗い出す（Java/Groovy: 4段階（直接 + 間接 + getter/setter経由）、C/Pro*C/Kotlin/C#・VB.NET/PL/SQL/TypeScript・JS/Python/Perl: 直接参照 + 定数経由のクロスファイル間接参照、Shell/SQL: 直接参照 + 同一ファイル内変数代入の間接参照）
- **使われ方の自動分類**: 言語ごとの使用パターンを自動判定し、人間の目視確認コストを大幅に削減する（Javaは javalang AST解析 + 正規表現フォールバック、その他はすべて正規表現分類）
さらに、その結果の横にひと目見ただけでそれがどのように使われているのかがわかる単位での出力があれば良い。

### プロダクトビジョン
Java / C / Pro*C / SQL / Shell / Kotlin / PL/SQL / TypeScript・JS / Python / Perl / C#・VB.NET / Groovy が混在するプロジェクト内の特定の文言（文字列値・コード値）を転用・削除しようとする開発者が、「この値は今どこでどのように使われているか」を完全に把握できるようにする。grepだけでは拾えない定数経由・getter / setter 経由の間接的な使用箇所まで追跡し、TSV形式で一覧化することで、数万行の手作業確認を不要にする。

### 目的
- grep結果4万行規模の手作業目視確認を自動化する。grep対象の資源は合計で60GB以上あるため、パフォーマンス面の考慮も重要となる。
- 言語ごとの使われ方パターンに自動分類し、影響範囲の把握を容易にする。さらに、その結果の横にひと目見ただけでそれがどのように使われているのかがわかる単位での出力があれば良い。
- 直接参照だけでなく間接参照（定数/変数/getter/setter経由）まで追跡し、転用・削除時の取りこぼしをゼロにする
- TSVを出力し、チームでの調査共有を容易にする

## ターゲットユーザー

### プライマリーペルソナ: Java・C・Pro*C混在レガシーシステムのリファクタリング担当者

- 大規模プロジェクト（数千〜数万ファイル、Java/C/Pro*C/SQL/Shell混在）を保守・改修している
- 特定の文字列値・コード値（ビジネスコード・ステータス値など）の意味を変更・廃止したい
- その文字列が現在どこでどのように使われているかを全件調査する必要がある
- **現在の課題**: grepで4万行ヒットしても、1行ずつ確認するのは現実的でない。また定数経由の間接使用はgrepでは拾えず、見落としが発生するリスクがある。（IDEなどで呼び出し改装検索をすれば同様のことは可能かもしれないが、それは大変なので避けたい）
- **期待する解決策**: 全件を自動分類したTSVを受け取り、Excelでフィルタしながら方針を決めたい。そのための助けとして、ひと目見ただけで理解できる粒度の記載の出力も横側に並んでいると嬉しい。
- **典型的なワークフロー**: grepを手動実行 → input/に配置 → ツール実行 → output/のTSVをExcelで確認
